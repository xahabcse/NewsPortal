## Working Rules (plan-first · autonomy · identity · testing · audit)

1. **Plan first, then build.** For any non-trivial task, present a short plan and get the user's approval **before** starting implementation. Never start coding on a substantial change without an approved plan. (Quick, obvious one-liners and pure investigation don't need a formal plan.)
2. **Once approved, run freely — ask only before deleting.** After the plan is approved, work end-to-end without pausing for permission on normal commands (PowerShell, cmd, git, gh, grep, npm/npx/node, curl, wrangler) and on Read/Write/Edit/Glob/Grep/WebFetch — this is wired in `.claude/settings.local.json` (broad `allow` + delete-only `ask`). **The one hard gate: confirm with the user before DELETING any file or data** (`rm`, `Remove-Item`, `del`, `rmdir`, `git clean`, `git reset --hard`, `git rm`, `find -delete`, etc.) — even if a delete path slips past the `ask` rules, stop and ask first.
3. **Identity for forms / configs / placeholder data.** When a name or email is required anywhere (config field, form, seed/test data, sample author), use **`Md Sahabuddin Hossain`** / **`sujoncep@gmail.com`**. This is NOT about the git commit author (leave git config as-is). For live mail-service testing, the global "ask which address first, never auto-pick" rule still applies.
4. **After any feature or bug fix: build locally, test on LIVE.** Run a local build to catch type/build errors (`npm --prefix src/NewsPortal.Client run build`, and `npm --prefix src/NewsPortal.Api.Hono run typecheck` for API changes), but do the actual behavioural verification on the **deployed/live site via MCP Playwright** — at mobile **and** desktop widths — not only on localhost. Delegate to the `ui-tester` agent or `/ui-test`.
5. **Playwright artifacts go in `.playwright-mcp/`.** All MCP Playwright data — screenshots, console logs, snapshots — must be saved under `.playwright-mcp/` (gitignored, registered as an additional working directory).
6. **After commit/push: monitor CI/CD, then hard-reload and test live.** Pushing to `dev` triggers the pipeline — don't walk away. Watch it (`gh run watch` / `gh run list --branch dev`) and confirm each stage actually **succeeds**: `ci-dev` build → auto-merged `dev→main` PR → `deploy-api-hono` (if `src/NewsPortal.Api.Hono/**` changed) and/or `deploy-client-pages` (if `src/NewsPortal.Client/**` changed). If any stage fails, fix it before calling the task done. Once live has the new deploy, **hard-reload / bypass cache first** — NewsPortal is a PWA, so the service worker + immutable asset caching can serve the *old* build until a hard reload (unregister SW + clear caches / cache-bust) — and only then run the live Playwright test.

### Git Commit Rules

Commit messages must be **clean — zero AI attribution**:

- ❌ Never add a `Co-Authored-By:` / `Co-authored-by:` trailer.
- ❌ Never add `Generated with` / `Generated by` / "🤖 Generated with Claude Code" or any similar attribution — in commits **or** PR bodies.

Write only the real change description (Conventional Commits style: `feat()`, `fix()`, `docs()`, `chore()`). Attribution is also disabled at the tooling level via `attribution.commit`/`attribution.pr` in `.claude/settings.local.json`.

## Code Style

- **Booleans:** for a genuine boolean, prefer `if (isSucc === false)` over `if (!isSucc)`. Do **not** apply `=== false` to null/undefined existence checks (`if (!row)` stays) — `undefined === false` is `false` and would break the logic.
- **Hono error handling:** one central `onError` maps `HTTPException` → its own response, `SyntaxError` (malformed/empty JSON body) → `400`, everything else → `500`. Don't wrap every route in try/catch just for body parsing.
- **Dates:** the user must **never** see a wrong date. Bengali/Hijri/Gregorian conversion lives in `src/NewsPortal.Client/src/utils/dateLocale.ts` (Bangladesh 2019 revised Bengali calendar; Hijri via `Intl` `islamic-umalqura`). Verify any date logic against known boundaries.
- Readable code, semantic variable names, comments only where the *why* is non-obvious.

Professional, editorial dark/light UI. New UI should speak this vocabulary:

|--------------------------------------|------|
| `background` / `foreground` | Page background / primary text |

## Architecture — two stacks

NewsPortal ships **two** backends. Know which one you're touching.

### A. Live production — Hono Worker + Cloudflare (current default)

Everything serving `news.xahabcse.me` today.

- **Worker:** `src/NewsPortal.Api.Hono/` — Hono + TypeScript on the **Cloudflare Workers runtime** (NOT Node.js). Worker name `newsportal-api-hono`.
- **Data:** Cloudflare **D1** (SQLite, binding `DB`, database `newsportal`) + **KV** (binding `CACHE_KV`) for response caching.
- **Vars** (`wrangler.toml`): `ADMIN_EMAIL`, `CORS_ORIGINS`, `PUBLIC_BASE_URL`, `CLOUDINARY_CLOUD_NAME`, `CLOUDINARY_API_KEY`.
- **Secrets** (`wrangler secret put`): `JWT_SECRET`, `GOOGLE_CLIENT_ID`, `GEMINI_API_KEY`, `CLOUDINARY_API_SECRET`.
- **Cron Triggers:** `*/5 * * * *` → `runScheduledFetch` (ingest due sources + inline body) · `*/30 * * * *` → `runBodyBackfill` (fill NULL-body articles of recent days). Both in `src/jobs/fetch-news.ts`, distinguished by `event.cron`; the scheduled handler also prunes logs.
- **AI:** Google **Gemini 2.5 Flash Lite** (`lib/gemini.ts`) — summarize / translate / sentiment, with graceful fallback.
- **Images:** **Cloudinary** (`lib/cloudinary.ts`) — remote-URL upload to the CDN.
- **Logging:** central `app_logs` table via `lib/logger.ts` (request / audit / extraction / client_error); viewer at `/admin/logs` (SuperAdmin).
- **Routes:** under `/api/v1/...` — `auth, news, feed, sitemap, robots, bookmarks, readhistory, reactions, reports, comments, newssources, fetchjobs, admin, adminarticles, usermanagement, analytics, images, ai, sse, logs` + `health`.
- **Migrations:** `src/NewsPortal.Api.Hono/migrations/NNNN_*.sql` (`0001_init.sql` full schema, `0002_app_logs.sql`). CI applies `--remote` on deploy.
- **Frontend:** `src/NewsPortal.Client/` — React 18 + TS + Vite + Tailwind, PWA, i18next (bn/en at `src/i18n/locales/`), SSE breaking-news ticker (`/api/v1/sse/breaking`; SignalR is legacy, disabled via `VITE_ENABLE_SIGNALR`). API base from `VITE_API_URL`.

### B. Self-host — .NET 8 + Docker (original full stack)

The original architecture; still in-repo and runnable, **not** auto-deployed (its `ci-cd.yml` is manual-only).

- **Projects:** `src/NewsPortal.Api` (ASP.NET Core 8 REST API), `src/NewsPortal.McpServer` + `src/NewsPortal.Scheduler` (Hangfire background fetch), `src/NewsPortal.Service` / `.Repository` / `.Core` (layered).
- **Data:** PostgreSQL (EF Core) + MongoDB (GridFS images) + Redis (cache) + Seq (logs).
- **Run:** `docker compose up -d --build` (full stack) or `docker compose -f docker-compose.dev.yml up -d` (DBs only) + `dotnet run`.
- **Config:** `.env` at repo root (see `README.md`).

> When a task touches the backend, confirm which stack is in scope. Default to the **Hono/Cloudflare** stack (it's what's live) unless the user says otherwise.

## Build & Deploy Flow

**Cloudflare (live) — git-push-to-deploy:**

1. `git push origin dev` → `ci-dev.yml`: typecheck `src/NewsPortal.Api.Hono` + build `src/NewsPortal.Client`, then auto-create + auto-merge a `dev→main` PR.
2. On `main`, path-filtered:
   - `deploy-api-hono.yml` (paths `src/NewsPortal.Api.Hono/**`) — `wrangler d1 migrations apply newsportal --remote` → `wrangler deploy --minify`.
   - `deploy-client-pages.yml` (paths `src/NewsPortal.Client/**`) — `npm run build` (with `VITE_API_URL`) → `wrangler pages deploy dist --project-name newsportal-client --branch main`.

Use `/deploy` for the full flow. **Docker/.NET:** `ci-cd.yml` is `workflow_dispatch`-only; deploy via `docker-compose.prod.yml`.
